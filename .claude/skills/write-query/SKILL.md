# /write-query - 编写优化的 Hive SQL

> 如果你看到不熟悉的占位符或需要查看已连接的工具，请参阅 [CONNECTORS.md](../../CONNECTORS.md)。

根据自然语言描述编写 Hive SQL 查询，遵循 Hive/HiveQL 最佳实践。

## 工作流程

### 第一步：理解需求 + 指标匹配

解析用户的描述，识别以下要素：

- **输出列** — 结果应包含哪些字段
  - 维度列：如年份、网点、主体等，用于 GROUP BY
  - 指标列：如销量、收入金额等，需要聚合函数（SUM/COUNT/AVG）
- **筛选条件** — 哪些条件限制了数据（时间范围、渠道类型、产品分类、状态）
- **聚合操作** — 指标列是否需要 SUM/COUNT 等聚合（金额/销量类字段通常需要）
- **连接** — 是否需要关联其他表（收入表、维表）
- **排序** — 结果如何排序
- **限制** — 是否有 top-N 或采样要求

**关键：同时在指标字典中查找匹配的标准指标**
- 加载 `references/metric_table_map.md` 查找是否有对应的标准指标
- 如果找到匹配指标，引用 `指标字典/{业务板块}/{业务分类}.md` 中的**技术口径 SQL** 作为参考
- 确保编写的 SQL 与指标字典中的口径定义保持一致

**指标匹配结果展示**
匹配到指标后，简化为：
```
已匹配指标：主宽入网数、129+宽带入网数（来源：全业务资料表）
口径来源：指标字典 - 基本面/宽带
```
直接询问用户："是否匹配？有遗漏的指标吗？"

### 第二步：找核心表

根据业务需求找表，而不是按字段匹配找表。

1. 参考 `references/TABLE_INDEX.md` 按业务主题定位表
2. 如果需求涉及特定指标，先查 `references/metric_table_map.md` 确定该指标对应的 CDAP 表
3. 优先使用指标字典中 `CDAP生产流程` 字段指向的表

### 第三步：加载表字段

加载表结构文档，确认字段映射、分区键、聚簇键等信息。

- 表结构文档位置：`references/tables/{序号}_{表名}.md`
- 表索引位置：`references/TABLE_INDEX.md`

### 第四步：匹配筛选字段

将需求中的筛选条件映射到具体列名。

### 第五步：编写查询

遵循以下最佳实践：

**优先使用简单单SELECT语句**
- 大多数查询用一个SELECT即可完成
- 只有在以下情况才使用CTE：
  - 需要多次复用同一子查询
  - 需要多维度交叉分析
  - 确实有多个逻辑步骤

**结构：**
- 当查询有多个逻辑步骤时，使用 CTE（WITH 子句）提高可读性
- 每个逻辑转换或数据源对应一个 CTE
- 为 CTE 取描述性名称（如 `daily_signups`、`active_users`、`revenue_by_product`）

**CASE WHEN 写法规则**
- 统计**用户数** → `COUNT(CASE WHEN 条件 THEN serv_id END)`
- 统计**金额/积分** → `SUM(CASE WHEN 条件 THEN 字段 ELSE 0 END)`
- 注意：统计用户数时不要用 `SUM(CASE WHEN 条件 THEN 1 ELSE 0 END)`，要用 COUNT

**性能：**
- 生产环境查询绝不使用 `SELECT *`——只指定需要的列
- 尽早筛选（将 WHERE 子句尽可能靠近基表）
- 善用分区过滤器（分区字段放在 WHERE 条件左侧，如 `dt = '2024-01-01'`）
- 大结果集子查询优先使用 `EXISTS` 而不是 `IN`
- 使用适当的 JOIN 类型（内联接正确时不要用左联接）
- 可以用 JOIN 或窗口函数时避免使用相关子查询
- 注意防止爆炸式连接（多对多）
- 考虑使用 `DISTRIBUTE BY` 和 `SORT BY` 优化数据倾斜

**可读性：**
- 为非显而易见的逻辑添加注释解释"为什么"
- 使用一致的缩进和格式
- 为表取有意义的首字母别名（不要只用 `a`、`b`、`c`）
- 每个主要子句单独占一行

**Hive 特定优化：**
- 使用 `SORT BY` 进行局部排序（每个 reducer 内排序），`ORDER BY` 用于全局排序（单 reducer，性能开销大）
- 使用 `DISTRIBUTE BY` 控制数据分发到 reducer，`CLUSTER BY` 等于 `DISTRIBUTE BY + SORT BY`
- 利用窗口函数（如 `ROW_NUMBER()`, `RANK()`, `LAG()`, `LEAD()`）替代自连接
- 使用 `UNION ALL` 而非 `UNION` 除非需要去重
- 善用 Hive 特有函数：`CONCAT_WS()`, `COLLECT_LIST()`, `EXPLODE()`, `PERCENTILE_APPROX()` 等
- 对于大表JOIN，考虑启用 `MAPJOIN` 提示或利用 bucket tables 优化

### 第六步：展示查询

提供：

1. **完整查询**——带语法高亮的 SQL 代码块
2. **简要说明**——每个 CTE 或部分的作用（如果是简单单SELECT则说明筛选条件）
3. **指标对齐说明**——如果使用了指标字典中的标准指标，说明口径来源
4. **性能说明**（如有）——分区使用情况、数据倾斜风险、潜在瓶颈
5. **修改建议**——如何调整以适应常见变化（不同时间范围、不同粒度、额外筛选条件）

**维度明细默认不开启**：默认只输出汇总数据，如用户需要维度拆分（如按渠道/网格），再通过 UNION ALL 或单独查询添加

### 第七步：提供执行选项

如果已连接数据仓库，主动提出运行查询并分析结果。如果用户想自己运行，查询已准备好可复制粘贴。

## 指标字典使用指南

### 什么是指标字典？

指标字典包含 90 个标准业务指标的定义，包括：
- **业务口径**：自然语言描述的业务定义
- **技术口径**：对应的 SQL 查询逻辑
- **责任人**：业务口径和技术口径负责人
- **CDAP生产流程**：对应的数据表或报表

### 如何使用指标字典

1. **需求匹配**：用户说"收入"时，查找指标字典中所有收入类指标（收保率、客保率、台阶收入等）
2. **口径参考**：复制指标字典中的技术口径 SQL 作为编写参考
3. **字段溯源**：通过 `CDAP生产流程` 找到对应表，加载表结构文档

### 指标字典文件位置

```
references/
├── 指标字典.md              # 索引入口
├── metric_table_map.md      # 指标→表映射
├── 指标字典/
│   ├── 基本面/              # 基本面指标（移动、宽带、双线）
│   ├── 专题/               # 专题指标（积分、收入、宽带T+n等）
│   └── 战新/               # 战新指标（云业务、卫星等）
```

## 示例

**简单聚合：**
```
/write-query 统计最近 30 天各状态的订单数量
```

**带指标匹配：**
```
/write-query 帮我统计一下宽带入网数，特别是129+宽带的入网情况
```
→ 自动匹配指标字典中"主宽入网数"、"129+宽带入网数"，引用对应口径

**复杂分析：**
```
/write-query 留存率分析——按注册月份对用户分组，然后展示在注册后 1、3、6、12 个月仍活跃（至少有一次事件）的用户比例
```

**性能关键：**
```
/write-query 我们有一张 5 亿行按日期分区的事件表。找出最近 7 天事件数量排名前 100 的用户及其最近一次事件类型。
```

## 技巧

- 提供表名和分区字段——这能大幅提升查询效率
- 说明查询是否需要幂等性（可安全重复运行）还是一次性查询
- 对于 recurring 查询，说明是否需要参数化日期范围
- 如果数据量大，提示是否需要采样或分批处理
- 优先在指标字典中查找标准指标定义，确保口径一致
